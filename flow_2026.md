Công thức cơ bản: User => Query => Embdding => Vector Search => Prompt => LLM. 

Tuy nhiên, để hệ thống chạy "pro" với hàng chục nghìn đến hàng triệu văn bản trong môi trường production năm 2026, kiến trúc thực tế sẽ phức tạp và tách biệt hơn nhiều.

Dưới đây là "bản đồ" kỹ thuật tóm gọn nhất để bạn hình dung: 

1. Kiến trúc phân tách (Separated Pipelines)
Hẹ thống chuyên nghiệp không chạy chung một luồng. Nó chia làm hai Pipeline riêng biệt: 
- Ingestion Pipeline (Async/ETL): Xử lý văn bản thô, làm sạch, chunking. tạo embedding và đẩy vào Vector DB. Thường dùng các queue như Kafka/ RabbitMQ để handle tải lớn. 
-Retrieval Pipeline (Low latency): Tiếp nhận query, rewrite query (HyDe hoặc Multi-query), tìm kiếm và trả kết quả trong <2 giây.

2. Chiến lược xử lý dữ liệu (The Secret Sauce)
Với hàng chục nghìn văn bản, "Fixed-size chunking" (cắt theo dộ dài đoạn) là sai lầm phổ biến nhất gây mất ngữ nghĩa. 
- Semantic Chunking: Sử dụng model (như BERT-based) để xác định điểm ngắt câu/ đoạn dựa trên sử thay đổi về consine similarity thay vì số lượng từ.
- Hybrid Retrieval: Luôn kết hợp Dense Retrival (Vector search) với Sparse Retrieval (BM25/ Keywork search). Dense tốt cho ngữ nghĩa, Sparse cực tốt cho từ khoá chuyên ngành, mã lỗi hoặc tên riêng. 
- Rẻanking (Cross-Encoder): Bước này cực kỳ quan trọng. Vector search tìm ra Top-K (ví dụ 100 kết quả), sau đó đẩy qua một model Reranker (như BGE-Reranker hoặc Cohere v3) để xếp hạng lại lấy Top-5 thực sự liên quan nhất. Nếu cần thiết có thể train cả model Reranker cho phù hợp với tác vụ đang thực hiện. 

3. Scaling & Production Performance 
Khi số lượng văn bản và người dùng tăng lên, hệ thống cần các kỹ thuật "hạng nặng":
- Vector DB Sharding & Partitioning: Chia nhỏ index theo metadata (ví dụ: tenant_id, department_id, year). Điều này giúp thu hẹp không gian tìm kiếm từ 10.000 xuống còn 500-1000 văn bản ngay từ bước lọc (pre-filtering).
- Quantization (Scalar/Production): Nén vector từ float32 xuống int8 hoặc thấp hơn để giảm RAM và tăng tốc độ scan mà không mất quá nhiều độ chính xác (Recall@K).
- Semantic Cache: Lưu lại các cặp (Query Vector, LLM Response). Nếu một query mới có similarity > 0.95 với query cũ, trả kết quả ngay lập tức, bỏ qua cả Retrieval và LLM inference (tiết kiệm chi phí và cực nhanh).
- HNSW Indexing: hầu hết các hệ thống pro dùng thuật toán đồ thị HNSW để tìm kiếm lân cận (AN N) thay vì scan toàn bộ (Flat search). 

4. Công nghệ "SOTA" năm 2026. 
- MRL: cho phép bạn cắt ngắn chiều vector (ví dụ từ 1536 xuống 256) mà vẫn giữ được độ chính xác cao, giúp linh hoạt giữa tốc độ và độ chính xác. 
- GraphRAG: Thay vì chỉ tìm văn bản rời tạc, hệ thống xây dựng Knowledge Graph để hiểu mối quan hệ giữa các thực thể (Entity). Rất hiệu quả khi hỏi các câu "tổng hợp" (ví dụ: "Tổng hợp các rủi ro pháp lý của tất cả các hợp đồng trong năm qua").
- Latte Interaction (ColBERT): Thay vì nén cả đoạn văn vào 1 vector, ColBERT giữ lại vector cho từng token. Độ chính xác cực cao nhưng tốn storage hơn. 

5. Tech Stack
- Vector DB: Qdrant (mạnh về filtering), Milvus (khả năng scale lớn), hoặc PGVector (nếu muốn dùng Postgres).
- Framework: LlamaIndex (mạnh về dât connector) hoặc LangGraph (cho Agentic RAG). 
- Evaluation: Ragas hoặc ARES. Đừng bao giờ tin vào cảm giác "đọc thấy đúng", phải đo bằng các chỉ số: Faithfulness, Answer Relevance, và Context Precision.
